# 上线验收清单

建议用一个独立钉钉测试应用、测试数据库和至少 1 个 HR + 3 个委员 + 1 个名单外账号执行。每项保存截图/请求 ID/审计记录，但截图不得包含真实密钥或无关个人信息。

## A. 构建与环境

- [ ] `npm ci`、`npm run lint`、`npm run typecheck`、`npm test`、`npm run build` 全部成功。
- [ ] 生产环境 `DINGTALK_MOCK_ENABLED=false`，访问 demo 登录被拒绝或不存在。
- [ ] 缺少数据库、会话密钥或钉钉关键凭证时应用 fail-fast，而不是带错误配置继续运行。
- [ ] `GET /api/health` 正常；响应不含数据库 URL、密钥、用户或投票数据。
- [ ] HTTP 自动跳 HTTPS；证书链、Cookie Secure 和代理协议识别正确。
- [ ] 真实 `organization.json` 位于仓库/镜像之外，权限受控；生产没有运行 `db:seed`。
- [ ] 迁移后、首次登录前的 `db:provision` 输出至少 1 名 HR、学术 10 人、技术 9 人，数据库实际有效名单一致。

## B. 钉钉登录与角色

- [ ] 钉钉内打开 H5 可免登；临时授权码不可重复使用。
- [ ] HR 进入管理端；委员只进入委员视图；名单外员工不能投票。
- [ ] 伪造姓名、角色、dingtalkUserId 或修改前端状态不能越权。
- [ ] 应用可见范围、域名和身份权限已经租户管理员核验。
- [ ] 会话过期后能安全重新登录，退出/撤权后旧会话失效。
- [ ] 示例占位 userId、错租户 userId 和重复 userId 不能通过生产初始化/登录验收。

## C. 发起投票

- [ ] 必填投票标题、结构化人选姓名、委员会和截止时间。
- [ ] 每场只创建一位人选，固定显示通过/不通过/弃权。
- [ ] 发起时固化本场委员名单；后续调整委员会成员不改变历史应投数。
- [ ] 截止时间边界、时区和 `Asia/Shanghai` 页面显示正确。
- [ ] 重复点击/网络重试不会创建两个相同场次（或界面明确提示）。

## D. 提交与修改

- [ ] 通过且意见空：拒绝；不通过且意见空：拒绝；弃权且意见空：成功。
- [ ] 每名委员每场只有一张当前有效票。
- [ ] 截止/关闭前可修改；统计立即按新票更新；旧版本完整保留。
- [ ] 两个页面并发修改不会静默覆盖历史，结果符合版本/事务规则。
- [ ] 名单外、其他委员会成员、HR 都不能代替委员投票。
- [ ] 到达截止时间但 cron 尚未运行时，提交仍被拒绝。

## E. 可见性与统计

- [ ] HR 无需投票即可看到 `已投/应投`、未投名单、三项票数和比例。
- [ ] 委员在投票期间看不到汇总、未投名单、他人选择或意见；直接调用 HR API 返回 403。
- [ ] HR 可查看记名明细和修改历史；普通委员不能读取。
- [ ] 已投为 0 时比例均为 0%；其他情况下三项票数合计等于已投人数。
- [ ] 历史修订不重复计票；系统不显示自动“通过/未通过”结论。

## F. 关闭和自动关闭

- [ ] HR 可在未全员时手动关闭，二次确认后所有后续提交/修改被拒绝。
- [ ] 手动关闭记录操作者、时间和原因。
- [ ] 使用错误/缺失 `x-maintenance-secret` 调用维护接口失败且不泄密。
- [ ] 正确调用 `POST /api/internal/maintenance/close-expired` 只关闭到期的 OPEN 场次，可重复调用。
- [ ] cron 连续失败触发告警；恢复后补关过期场次。

## G. 一键私聊提醒

- [ ] `POST /api/polls/:id/remind` 只允许 HR。
- [ ] 服务端重新计算未投名单；前端篡改收件人无效。
- [ ] 每位未投委员收到私密消息，已投委员和名单外人员不收到。
- [ ] 消息只含必要信息和正确深链，不含统计、他人身份或意见。
- [ ] 重复点击受频率限制；部分失败显示真实结果并可审计。
- [ ] 实际钉钉权限、配额和点对点到达效果已由租户管理员验收。

## H. 检索、导出与长期留存

- [ ] 可按结构化人选姓名、标题和时间范围检索历史场次。
- [ ] Excel 导出的场次信息、应投/已投、统计、记名明细与页面一致。
- [ ] 以 `= + - @` 开头的意见不会在 Excel 中执行公式。
- [ ] 只有 HR 可导出，导出行为有审计；下载链接不可长期公开。
- [ ] 关闭场次、重启应用和超过钉钉消息保留时间都不影响 PostgreSQL 历史记录。

## I. 备份、恢复与运维

- [ ] 每日备份生成 `.dump` 和 `.sha256`，失败会告警；日志不含连接串/口令。
- [ ] 备份已复制到加密异地存储，权限和保留策略已审批。
- [ ] 在隔离数据库成功校验哈希、`pg_restore` 并完成数据/权限抽查。
- [ ] 记录本次恢复的 RPO/RTO；结果达到单位目标。
- [ ] 应用、数据库、自动关票、催票和备份均有负责人和监控看板。
- [ ] 发布回滚演练通过；数据库迁移有兼容和回滚说明。
