# 钉钉企业内部 H5 应用配置

本文只描述接入原则。钉钉开放平台的菜单、权限点名称、审批要求和接口可用范围可能随版本、租户版本及管理员策略变化；上线前必须由本企业的钉钉主管理员在当前开放平台后台逐项核对。不要根据本文猜测 CorpId、应用 ID、密钥、robotCode 或权限标识。

## 1. 创建与归属

1. 由企业主管理员或经授权的开发管理员创建“企业内部应用”，应用归属必须是目标企业。
2. 启用 H5/微应用入口，配置移动端主页为生产 HTTPS 地址，例如 `https://<正式域名>/`。
3. 设置可见范围：至少包括两委会委员和 HR；建议按最小范围授权，不默认全员可见。
4. 记录后台真实显示的 CorpId、ClientId/AppKey、ClientSecret/AppSecret。密钥只进入服务端 Secret Manager。
5. 应用发布前完成企业内部审批、版本发布和可见范围确认。

## 2. 域名、回调和 HTTPS

- `DINGTALK_APP_BASE_URL` 必须是用户实际访问的公网 HTTPS origin，不带密钥，通常不带末尾路径，例如 `https://vote.example.com`。
- 将同一正式域名按当前控制台要求配置到“应用首页/回调域名/授权域名/JSAPI 安全域名”等相关位置。后台没有的字段不要自行新增；后台要求的字段不要遗漏。
- H5 登录由前端取得临时 `authCode`，再调用本服务 `POST /api/auth/dingtalk`。
- 桌面浏览器登录使用钉钉 OAuth 授权码流程，控制台必须登记与 `DINGTALK_WEB_REDIRECT_URI` 完全一致的地址。服务器回调路由为 `GET /api/auth/dingtalk/web/callback`。SSH 真实数据联调可临时使用本机私有局域网地址，并由启动脚本显式开启 `DINGTALK_WEB_ALLOW_INSECURE_REDIRECT=true`；其它非 HTTPS 地址会被拒绝。
- 浏览器登录需要个人委托权限 `Contact.User.Read`。授权范围包含 `openid corpid`；服务端会校验所选组织，并将 `unionId` 换成企业内 `userId` 后再匹配系统名单。
- 浏览器内的委员会成员选择器由服务端读取子部门和部门用户列表。需在钉钉开放平台为应用审批相应的企业通讯录读取权限，并确保应用通讯录可见范围覆盖 HR 需要维护的部门；权限不足时界面会明确提示，不能退化为手工填写用户 ID。
- 证书必须由受信 CA 签发且链完整；禁止生产使用自签名证书。反向代理需保留正确的 `Host` 和 `X-Forwarded-Proto=https`。
- 不要把 localhost、临时隧道域名或测试域名留在生产白名单。测试应用与生产应用应分开。

## 3. 免登身份链路

H5 只能把钉钉客户端产生的一次性临时授权码发给服务端：

```http
POST /api/auth/dingtalk
Content-Type: application/json

{"authCode":"<临时授权码>"}
```

服务端使用内部应用凭证调用当前官方身份接口，取得稳定用户标识，然后匹配本地 HR/委员授权。`GET /api/session` 只把可公开的 CorpId 返回给 H5 以调用 JSAPI；CorpId 不使用 `NEXT_PUBLIC_*` 构建变量，应用密钥也绝不返回浏览器。需要管理员核验并审批“免登/读取当前用户基础身份”所需的实际权限点与可见范围。

重要约束：

- 不接受前端传入的姓名、角色或 dingtalkUserId 作为最终身份。
- 不把临时授权码、access token、AppSecret 写日志。
- 如果用户不在应用可见范围、本地委员名单或 HR 授权表中，默认拒绝访问。
- 稳定用户标识是数据库关联键；姓名只作展示和历史快照，改名不能产生新账号。

### 首次建立本地授权映射

钉钉应用“可见范围”不等于本系统 HR/委员权限。生产数据库迁移后、首次登录前，复制 `config/organization.example.json` 到仓库之外的受控路径，填写从本企业钉钉管理流程取得的真实稳定 userId，然后执行：

```bash
npm run db:provision -- --file /secure/path/organization.json --confirm
```

脚本要求至少 1 名 HR、学术委员会恰好 10 人、技术委员会恰好 9 人，并拒绝重复 userId。名单文件含人员身份信息，需由 Secret/配置管理系统或权限受控目录保存，不提交仓库，也不要发到群聊。

该命令用于空生产库的首次冷启动；相同名单重复执行为更新式写入，但当前脚本不会因为某人从配置中消失就自动停用旧成员。后续换届、离职或委员替换必须走经审批的成员变更流程，并在新场次发起前复核当前有效人数；历史场次的名单快照不能改写。

## 4. 私聊催票能力

HR 点击催票后，服务端从本场名单快照计算未投人员，逐人发送私密提醒。不同租户可用的实现可能是企业内部应用工作通知、应用机器人单聊或当前开放平台提供的另一种点对点消息能力。

上线前由管理员和研发共同确认：

- 当前企业内部应用能使用哪一种“仅目标用户可见”的消息接口；
- 该接口的实际权限点、应用可见范围、发送频率/配额和内容限制；
- 是否需要 `robotCode`，以及它与当前应用是否正确关联；
- 消息能否在目标委员的钉钉客户端稳定到达，并打开正确的投票深链；
- 失败、限流、用户离职/不可见时的错误处理方式。

不得为了绕过私聊权限而退化成在委员会群里公布未投名单。若租户暂未授权任何点对点能力，应在管理端明确显示“提醒未发送”并保留失败审计，不能假装成功。

建议提醒只包含：投票标题、人选姓名、截止时间和打开投票的应用内链接；不要包含其他委员的选择、意见或实时统计。

## 5. 环境变量：mock 与真实钉钉

| 变量 | 本地演示 | 生产 |
| --- | --- | --- |
| `DINGTALK_MOCK_ENABLED` | `true` | 必须为 `false` |
| `ALLOW_INSECURE_PRODUCTION_MOCK` | `false` | 必须为 `false`；仅隔离内网演示环境可临时启用 |
| `DINGTALK_CORP_ID` | 留空 | 后台实际值 |
| `DINGTALK_CLIENT_ID` | 留空 | 后台实际值 |
| `DINGTALK_CLIENT_SECRET` | 留空 | Secret Manager 注入 |
| `DINGTALK_ROBOT_CODE` | 留空 | 仅所选消息能力实际要求时填写 |
| `DINGTALK_APP_BASE_URL` | 可用本机 URL | 正式 HTTPS origin |

应用启动时建议执行 fail-fast 校验：生产环境若 mock 为 true、关键凭证为空、base URL 不是 HTTPS，直接拒绝启动。

## 6. 联调顺序

1. 先在本地 mock 模式完成投票业务、权限和边界测试。
2. 创建独立的钉钉测试应用，限制在少量测试账号可见。
3. 配置测试 HTTPS 域名，验证 H5 内免登和会话 Cookie。
4. 分别用 HR、名单内委员、名单外员工验证服务端授权。
5. 在普通桌面浏览器中验证委员会成员选择器可浏览部门、分页加载用户并跨部门选择；再在钉钉客户端内验证原生选择器快捷入口。
6. 申请并验证点对点消息能力；故意撤销权限，确认系统能显示真实失败。
7. 验证应用首页、提醒深链、过期会话重新登录。
8. 由管理员复核生产权限和可见范围后，再切换生产应用凭证。

## 7. 上线前管理员确认单

- [ ] 应用类型和归属企业正确，测试/生产应用分离。
- [ ] H5 首页、授权/安全域名及实际登录方式与当前控制台一致。
- [ ] 应用只对必要人员/部门可见。
- [ ] 免登身份权限已审批，接口实际调用成功。
- [ ] 子部门与部门用户读取权限已审批，普通浏览器中的企业通讯录选择器已实测。
- [ ] 点对点提醒所需能力、权限、配额和 robotCode（如需）已实测。
- [ ] 没有把密钥放在前端 bundle、仓库、截图或群消息中。
- [ ] 离职、转岗、委员会换届时有可执行的权限更新流程；历史场次名单快照不被改写。

> 说明：本仓库未连接目标企业租户后台，无法替代租户管理员核验。本文刻意不列举可能过期或不适用于本租户的权限 scope 名称和控制台菜单路径；实际配置必须以当前钉钉开放平台后台、官方文档和管理员审批结果为准。
