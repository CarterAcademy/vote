# 安全与隐私检查清单

投票选择、评审意见、未投名单和人员身份属于敏感业务数据。以下控制既适用于代码评审，也适用于上线审批。

## 身份与授权

- [ ] 生产强制 `DINGTALK_MOCK_ENABLED=false` 且 `ALLOW_INSECURE_PRODUCTION_MOCK=false`；mock 登录路由在生产不可访问。后者仅允许隔离的内网演示环境使用优化后的 production build，不得用于真实业务部署。
- [ ] 临时 `authCode` 只发往本服务 HTTPS endpoint，单次使用，不记录。
- [ ] AppSecret、数据库口令、会话密钥、维护密钥只在服务端 Secret Manager 中。
- [ ] 会话 Cookie 使用 `HttpOnly`、`Secure`、合理 `SameSite`、短有效期和轮换策略。
- [ ] 每个 API 在服务端鉴权；HR 能力使用显式角色检查，不能只隐藏按钮。
- [ ] 委员资格按“场次名单快照”判断，名单外员工即使能打开应用也不能投票。
- [ ] HR/管理员离职或转岗时及时撤权；至少每月复核一次。
- [ ] 首次 `organization.json` 存在仓库外的受控路径，权限最小化且不进入镜像、日志、工单正文或群聊。
- [ ] 生产迁移后、首次登录前运行 `db:provision`；输出确认至少 1 名 HR、学术 10 人、技术 9 人，且未运行 demo seed。

## 投票机密性与完整性

- [ ] 委员接口和页面不返回实时票数、比例、未投名单、他人身份/选择/意见。
- [ ] 当前票有 `(poll_id, voter_id)` 唯一约束。
- [ ] 通过/不通过意见必填；弃权意见可空；前后端校验一致。
- [ ] 修改只允许在 OPEN 且未到截止时间时进行，并在同一事务保留修订历史。
- [ ] 截止后的提交即使自动关票 cron 延迟也会被拒绝。
- [ ] 手动关闭允许未全员，但要求 HR 二次确认并写审计。
- [ ] 统计只计算每人的当前有效票，不把历史版本重复计票，也不自动下结论。
- [ ] 导出文件标明敏感级别，只提供给 HR；下载 URL 短时有效且不可猜测。

## 催票与钉钉消息

- [ ] 收件人完全由服务端计算，客户端不能提供任意用户 ID 列表。
- [ ] 只发送给尚未提交当前票的本场委员，不在群里公开未投名单。
- [ ] 有单场/单 HR/单位时间频率限制，重复点击不会骚扰用户。
- [ ] 发送内容不包含投票统计、其他委员信息或意见。
- [ ] 逐人记录发送结果；权限或限流失败不伪装成成功。

## Web 与接口安全

- [ ] 所有公网流量使用 TLS；正确处理可信反向代理头。
- [ ] 写接口具备 CSRF 防护（SameSite 之外再校验 Origin/CSRF token，按架构选择）。
- [ ] 设置 CSP、`frame-ancestors`、`X-Content-Type-Options`、Referrer-Policy 等响应头，并验证不会破坏钉钉 WebView。
- [ ] 对登录、投票、导出、催票和维护接口限流；请求体有大小限制。
- [ ] 输入用 schema 校验；数据库访问参数化；导出时防止 Excel 公式注入（以 `= + - @` 开头的用户文本需转义）。
- [ ] 错误响应不泄露 SQL、堆栈、内部 ID 映射、令牌或意见正文。
- [ ] `MAINTENANCE_SECRET` 与 `SESSION_SECRET` 不复用；维护接口只接受 POST、恒定时间比较并可叠加网络限制。
- [ ] 依赖锁文件提交仓库，CI 执行依赖漏洞和许可证检查；高危漏洞有修复时限。

## 数据库、日志和备份

- [ ] PostgreSQL 不对公网开放；应用使用独立最小权限账号和 TLS。
- [ ] 生产、测试、开发数据完全隔离；禁止把生产意见复制到开发环境。
- [ ] 业务记录没有通用硬删除入口；任何纠错/删除均需经批准并留审计。
- [ ] 审计日志采用追加写并限制访问；不写 Cookie、token、AppSecret、完整意见正文。
- [ ] 每日备份成功且有 SHA-256；备份加密、异地保存、限制删除权限。
- [ ] 每季度恢复演练并记录 RPO/RTO；备份失败和异地复制失败会告警。
- [ ] `BACKUP_RETENTION_DAYS` 和业务数据长期保留策略分别审批，不能误把备份清理当业务清理。

## 隐私治理

- [ ] 告知委员投票为记名、HR 可见明细、使用目的和保留策略。
- [ ] 只收集完成评审所需的身份、选择、意见和审计字段。
- [ ] HR 查看/导出明细遵循最小知情原则；导出后有存储、转发和销毁制度。
- [ ] 明确数据责任人、系统管理员、应急联系人和访问审批流程。
- [ ] 根据单位制度及适用法律确认“长期保存”的具体期限、合法依据和到期处置，不在代码中私自推定“永久”。

## 事件响应

- [ ] 发现越权、数据泄漏、票被篡改或密钥泄漏时，能立即停用相关凭证/会话并保留证据。
- [ ] 密钥轮换不会要求改写历史票；轮换步骤在预发布演练过。
- [ ] 审计可回答谁在何时发起、投票/修改、查看/导出、催票、关闭及操作结果。
- [ ] 安全事件后从只读副本核对当前票与修订历史，未经审批不直接修改生产记录。
